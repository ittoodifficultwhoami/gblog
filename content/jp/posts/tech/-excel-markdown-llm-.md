日本国内のビジネス環境において、Excelを方眼紙のように活用し、印刷を前提とした「神エクセル」形式で帳票を作成する慣習は、依然として根強く残っている。キーマンズネットの2020年調査では、約6割の企業がいまだにこの形式で帳票を作成している実態が明らかとなった。これが日本の労働生産性を押し下げる一因であり、OECD加盟38カ国中28位（2024年）という順位の一端を担っていることは否定できない。

2016年、当時行政改革担当大臣であった河野太郎氏が文部科学省へ「神エクセル」廃止を指示するなど、行政レベルでの標準化は進められてきた。しかし、現場では長年蓄積されたレガシーデータの存在が壁となり、機械判読可能なデータ形式（Machine-readable）への完全な移行には至っていない。多くの現場では「Excelでなければ承認プロセスが回らない」という運用上の制約が、システムの刷新を阻んでいるのが実情である。本稿では、ごっち氏が2026年3月20日にZennで公開した検証記事に基づき、方眼紙ExcelをMarkdown形式へ変換する手法の現在地を技術的視点から整理する。

### 技術的アプローチの比較分析

帳票データを構造化データへ変換する際、現在以下の3つのアプローチが検討されている。

| 手法 | 仕組み | メリット | デメリット |
| :--- | :--- | :--- | :--- |
| 直接抽出 | Pythonのpandas等でセルの座標を指定して抽出 | 変換精度が安定する | フォーマット変更ごとの保守コストが高い |
| AI-OCR活用 | OCRで視覚的に解析しテキスト化 | 非定型データに強い | 誤認識の修正に人手が必要となる |
| LLMベース | LLMを介してセル関係を推論・構造化 | 柔軟なデータ再構築が可能 | トークン消費コストと推論時間の増大 |

1. **直接抽出法によるアプローチ**
   Excelのセル位置を基準にデータをスクレイピングする手法である。特定の帳票テンプレートが完全に固定されている場合に有効だが、現場で勝手に列が挿入されたりレイアウトが変更されたりするだけで処理が破綻する脆弱性がある。

2. **LLMを活用した推論変換**
   ごっち氏の検証では、LLMを用いてExcel内の配置から意味的な階層構造を読み解く手法が提示されている。従来の静的な座標指定とは異なり、項目名と値の関係性をコンテキストとして保持できるため、Markdown形式への変換において高い再現性を実現している。これは、機械判読不可能な「見た目重視」のデータであっても、意味論的な抽出が可能になりつつあることを示唆する。

この技術的遷移は、単なるフォーマット変換ではなく、データ資産の再定義に近い。Markdownへの変換は、単にドキュメントとして管理しやすくするだけでなく、GitHub等でのバージョン管理やCI/CDパイプラインへの統合を可能にするため、長期的には業務自動化の基盤となる。

![Simple cartoon illustration: A comparison chart showing three robotic arms: one lifting a tiny cube (tool), one holding a heavy block (Docling), and one handling a glowing light bulb (LLM).](/images/-excel-markdown-llm-_0.png)

従来、Excelデータの抽出には `openpyxl` や `pandas` といったライブラリが利用されてきました。これらはセルの座標データに基づいて処理を行うため、結合セルや意図的な空行が混在する、いわゆる「方眼紙形式」のファイルでは、データの構造を正しく解釈できないという技術的課題がありました。

近年登場したAIベースの変換ツールおよびLLMを活用する手法との比較を以下に示します。

| ツール名 | 開発元 | 特徴 | リソース消費量(目安) |
| :--- | :--- | :--- | :--- |
| **MarkItDown** | Microsoft | オープンソース変換ツール | 約250MB / メモリ450MB |
| **Docling** | IBM | TableFormerによる細胞抽出 | 約1.1GB / メモリ1.8GB |
| **LLM (API利用)** | 各社 | 文脈理解による再構成 | トークン課金制 |

### 1. 専門ツールの特性
IBMが提供する **Docling** は、AIモデル「TableFormer」を搭載しており、表細胞の抽出精度は97.9%〜99%に達します。一方、2024年11月に公開されたMicrosoftの **MarkItDown** は、軽量かつ高速な変換を志向しており、開発環境やサーバー負荷に応じてこれらを使い分けるのが合理的です。特にDoclingはドキュメント解析のパイプラインとして完成度が高く、社内システムへの組み込みに適しています。

### 2. LLM活用の優位性と限界
LLM（GPT-4oやClaude 3.5 Sonnetなど）の真価は、表を単なる座標の集まりではなく「文脈」として解釈する点にあります。従来のプログラム変換では対処が困難だった、結合セルの論理的な意味の補完や、視覚的なレイアウトに応じた構造的なMarkdown変換が可能です。

しかし、実務への導入には以下の制約が伴います。

*   **トークン消費量:** スクリーンショット1枚あたり170〜1100トークンを消費します。大規模なドキュメントを一括処理する場合、Claude 3.5 Sonnetの200,000トークンやGPT-4oの128,000トークンといったコンテキストウィンドウの制限を考慮した設計が必要です。
*   **精度の不確実性:** 複雑なフィールドを含むデータセットでは、現行の最高性能モデルであっても精度は60〜80%程度に留まることが多く、最終的な品質担保のためにHuman-in-the-loop（人間の介入）を工程に組み込む必要があります。

### 推奨される変換フロー
1.  **事前フィルタリング:** 文書全体をAIへ投入する前に、メタデータを用いて不要なページを物理的に除外する。
2.  **構造変換:** 専門ツール（Docling等）で抽出された中間データに対して、LLMによる構造補正を行う。
3.  **バリデーション:** スキーマ定義に基づいたコード検証（Pydantic等）を行い、不適合データのみを人間が再確認する。

この段階的なプロセスを経ることで、AI特有のハルシネーション（誤情報）を抑えつつ、方眼紙形式の複雑なデータを実務レベルの精度で構造化データへ変換可能です。

![Simple cartoon illustration: A three-step assembly line: Step 1 a camera icon, Step 2 a text document icon, Step 3 a lightbulb representing the hybrid LLM output.](/images/-excel-markdown-llm-_1.png)

Excel方眼紙をMarkdown形式へ変換する際、既存のツールや単純なOCR処理では、レイアウトの崩れやセルの結合情報の欠落が課題となります。現在、実務において最も効率的かつ高精度とされる手法は、視覚情報とテキスト情報を統合する「ハイブリッドアプローチ」です。

### ハイブリッドアプローチの実行手順

1. 視覚情報の確保
Excelのセル構成や罫線の意図を言語モデルに認識させるため、対象シートのスクリーンショットを生成します。高解像度での保存が前提となります。

2. テキストデータの抽出
OCRの誤読リスクを排除するため、Excel上から直接コピーしたテキストデータを作成します。この数値情報が、変換後のMarkdownにおける正確性の根拠となります。

3. ハイブリッドプロンプトによる変換
画像情報でレイアウト構造を、抽出したテキスト情報で数値の整合性を保証する構成でLLMへ指示を出します。これにより、テーブル構造の再構築にかかる工数を最小化させることが可能です。

### 比較データと技術構成

| 手法 | 精度 | 構造保持率 | 実装負荷 |
| :--- | :--- | :--- | :--- |
| 単純OCR変換 | 低 | 低 | 低 |
| ルールベース変換 | 中 | 中 | 高 |
| ハイブリッド変換 | 高 | 高 | 中 |

この手法の有効性は、技術検証系メディア「データ利活用100本」（2026年3月22日配信、穂高いづる氏）においても、「GitHub Models APIを活用した変換構成」として提示されています。特に大規模な帳票データにおいて、セルの結合を伴う複雑な表組みをMarkdownの標準記法（| --- |）へ変換する際、後編集の手間を大幅に削減できる点が最大の利点です。

### 実務上の指針

Excel方眼紙のデータは単なる情報の羅列ではなく、罫線自体が意味を持つドキュメントです。変換過程において「数値の正しさ」を保持するためには、LLM単体での画像解析に依存せず、常にテキスト情報を補完データとして渡す設計が不可欠です。

今後は、OpenAIのo3-miniなどの推論型モデルを活用することで、複雑なセル結合を持つ帳票の階層構造化がさらに自動化される見込みです。現時点では、このハイブリッドアプローチを基盤としつつ、出力されたMarkdownの構文チェックを自動化するスクリプトを併用するのが、最も堅実なワークフローとなります。

![Simple cartoon illustration: A person with a checklist and a pen standing next to a computer screen, reviewing generated Markdown code to ensure quality.](/images/-excel-markdown-llm-_2.png)

Excel方眼紙の構造的な問題は、現在、LLMによる高度な解釈技術によって解消可能な段階にある。しかし、変換結果の正確性を完全に自動で担保することは依然として困難であり、現在は技術的な過渡期に位置している。

### 業務における実装の指針

1. 完全自動化の放棄と半自動化へのシフト
LLMを「自動生成ツール」と捉えるのではなく、「変換・整形アシスタント」として位置づける。出力に対して人間が最終確認を行うプロセスを業務フローに組み込むことが、実用上の前提となる。

2. 総務省指針に基づく標準化
総務省が策定した「統計表における機械判読可能なデータ作成に関する表記方法（15項目）」を準拠すべき規準とする。この指針に基づき、既存のExcel資産を可能な限りListObject（テーブル）形式へ整形してからLLMに読み込ませることで、変換精度は飛躍的に向上する。

3. Copilot製品と独自パイプラインの使い分け
Microsoftが提供する標準的なExcel Copilotは、セル結合や装飾を多用した方眼紙データの解釈には不向きである。複雑なレイアウトを処理する場合、APIを介してプロンプトやコンテキストを制御可能な「GitHub Copilot SDK」や、各社LLMのAPIを利用した独自の変換パイプラインを構築する必要がある。

### 技術的考察

方眼紙データからMarkdownやJSON形式へ移行する際、最も障壁となるのは結合セルによって破壊された「主キー」の概念である。多くの実務において、人間はセルの配置という視覚情報で列同士の関係性を把握しているが、LLMはモデルのトークン上限とアテンションの仕組み上、その構造を誤認しやすい。

「方眼紙Excelを撲滅する」という理想と、「既存のレガシーデータをいかに機械判読可能な形式へ移行するか」という実務の狭間で、LLMは極めて強力な橋渡し役を担う。各ツールの特性を正確に理解し、適切な精度管理の下で変換パイプラインを構築することが、現在のデータ利活用における現実的な最適解である。
